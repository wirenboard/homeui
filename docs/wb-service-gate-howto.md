# Закрыть свой сервис авторизацией Wiren Board (порт-гейт)

Сервис живёт на отдельном **порту**; nginx Wiren Board пускает на него только тех, кто
залогинен в веб-интерфейсе (homeui) с нужной ролью. Неавторизованного в браузере
перекидывает на форму входа, API-клиент получает `401`.

> Требует homeui с gate-сниппетами и поднятым лимитом `/auth/check` (см. «Состав»
> внизу). На более старом — сначала обновить homeui.

## Как это устроено

- Сервис слушает **только `127.0.0.1`** — снаружи невидим, единственный вход через
  nginx.
- На внешнем порту nginx делает `auth_request` к `/auth/check`: валидная сессия нужной
  роли → пускает, иначе → форма входа (браузер) или `401` (API).
- Короткий **редирект под путём** (`https://<хост>/open-<сервис>` → порт) даёт
  авто-возврат на сервис после логина и удобную относительную ссылку без порта.

## Конвенция портов

Внешний порт = **«2» перед внутренним**: `1880 → 21880`, `9000 → 29000` (для внутренних
портов < 10000). Внешний и внутренний **обязаны различаться** — иначе nginx на
`0.0.0.0:<порт>` перекрыл бы сам сервис на `127.0.0.1:<порт>`.

## Настройка

### 1. Сервис на loopback

Сервис должен слушать `127.0.0.1:<внутренний-порт>` (например `127.0.0.1:9000`), не
`0.0.0.0` — иначе порт обходит гейт.

### 2. Проверить, что внешний порт свободен

```bash
ss -ltn | awk '{print $4}' | grep -E '[:.]29000$'   # пусто = порт свободен
```

### 3. Файл гейта

`/etc/nginx/conf.d/<сервис>.conf` — поменять 4 отмеченные строки:

```nginx
server {
    listen 29000 ssl;                              # (1) внешний порт (= 2 + внутренний)
    server_name _;
    include /etc/nginx/snippets/wb-gate-tls.inc;

    location / {
        set $wb_role   admin;                      # (2) минимальная роль: admin | operator | user
        set $wb_return /open-my-service;           # (3) путь возврата (см. шаг 4)
        auth_request /wb_auth_check;
        error_page 401 = @wb_unauth;
        proxy_pass http://127.0.0.1:9000;          # (4) внутренний адрес сервиса
        include /etc/nginx/snippets/wb-gate-proxy.inc;
    }
    include /etc/nginx/snippets/wb-gate-authcheck.inc;
    include /etc/nginx/snippets/wb-gate-unauth.inc;
}
```

Роль работает **«эта и выше»** (`user < operator < admin`): `user` — любой
залогиненный, `admin` — только администратор.

### 4. Редирект под путём (рекомендуется)

`/etc/nginx/includes/default.wb.d/<сервис>-bounce.conf` — порт **должен совпасть** с
`listen` из шага 3:

```nginx
location = /open-my-service { return 302 https://$host:29000/; }
```

После логина homeui возвращает только на **свой origin** (другой порт отбрасывается как
open-redirect), поэтому `$wb_return` указывает на этот путь, а он уже 302-редиректит на
порт. Плюс это готовая относительная ссылка без хоста и порта. Без файла гейт работает,
но после логина пользователь окажется на дашборде homeui.

### 5. Проверить и перезагрузить

```bash
nginx -t && systemctl reload nginx
```

Если `nginx -t` ругается — исправить файл и **не** перезагружать: битый конфиг уронит
следующий reload вместе с веб-интерфейсом.

### 6. Пункт в меню «Интеграции» (опционально)

`/usr/share/wb-mqtt-homeui/custom-menu/<сервис>.json` — применяется без рестарта:

```json
{
  "id": "integrations",
  "children": [
    {
      "id": "my-service",
      "url": "/open-my-service",
      "isExternal": true,
      "openInNewTab": true,
      "title": { "ru": "Мой сервис", "en": "My Service" }
    }
  ]
}
```

`url` — тот же относительный путь из шага 4. Обновить страницу homeui — пункт появится.

## Обязательно проверить

1. **Сервис слушает только `127.0.0.1`** — иначе порт обходит гейт.
2. **Открывать сервис по тому же hostname, что и homeui** (cloud-DNS
   `<ip>.<serial>.ip.wirenboard.com`) — сессионная кука host-only; на другом имени её
   нет, и пойдёт бесконечный редирект.
3. **`nginx -t`** перед reload (шаг 5).
4. **Роль выбрать осознанно** — `user` пускает любого залогиненного.

## Как пользоваться

- Прямой вход: `https://<хост>:29000/`.
- Удобный: `https://<хост>/open-my-service` (или пункт меню) → редирект на порт.

## Ограничения

- **Облачный тоннель Wiren Board** пробрасывает только основной веб-порт (443);
  дополнительный порт недоступен — порт-гейт работает в локальной сети / при прямом
  доступе.
- **API-клиенты (скрипты)** аутентифицируются сессионной кукой: клиент логинится
  (`POST /auth/login`), получает куку `id` и шлёт её с запросами. Браузер получает
  редирект на форму, запрос без куки — `401`.

## Состав (со стороны homeui)

- Лимит `/auth/check` поднят до **1000** (`/auth/login` остаётся 30) — держит всплеск
  авторизованных подзапросов при открытии интерфейса без отдельного кэша на каждый
  сервис.
- Gate-сниппеты в `/etc/nginx/snippets/`: `wb-gate-tls.inc` (TLS внешнего порта),
  `wb-gate-authcheck.inc` (`auth_request` к `/auth/check`), `wb-gate-unauth.inc`
  (фоллоу-ап: навигация в браузере → форма входа, иначе `401`), `wb-gate-proxy.inc`
  (proxy-заголовки, срезает клиентские `X-WB-*`, проброс websocket).
- Поддержка во фронтенде: возврат на сервис после логина (`externalReturn`) и внешние
  пункты меню (`isExternal`).
