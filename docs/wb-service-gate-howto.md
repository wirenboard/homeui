# Закрыть свой сервис авторизацией Wiren Board (порт-гейт)

Черновик пользовательской инструкции + что для неё несёт homeui. Модель: сервис
живёт на отдельном **порту**, nginx Wiren Board пускает на него только тех, кто
залогинен в веб-интерфейсе (homeui), с нужной ролью. Неавторизованного в браузере
перекидывает на форму входа; API-клиент получает чистый `401`.

> **Работает с homeui, который уже несёт gate-сниппеты и поднятый лимит
> `/auth/check` (1000) — см. «Что несёт homeui» внизу. На более старом — сначала
> обновить homeui.**

---

## Как это устроено (коротко)

- Сервис слушает **только `127.0.0.1`** — снаружи он невидим, единственный вход в
  него это nginx.
- На внешнем порту nginx делает `auth_request` к homeui `/auth/check`: есть
  валидная сессия нужной роли → пускает, нет → форма входа (браузер) или `401`
  (API). Кэша нет — лимит `/auth/check` поднят до 1000 и сам держит всплеск
  запросов при открытии интерфейса.
- Дополнительно — короткий **редирект под путём** на том же origin
  (`https://<хост>/open-<сервис>` → порт). Он даёт: (1) авто-возврат на сервис
  после логина, (2) удобную относительную ссылку без порта (для меню, шаблонов,
  локального доступа).

## Конвенция портов

Внешний порт по умолчанию = **«2» перед внутренним**: сервис на `1880` → внешний
`21880`, сервис на `9000` → внешний `29000`. (Правило для внутренних портов
< 10000; иначе значение выйдет за 65535 — указать порт вручную.) Внешний и
внутренний **обязаны различаться**: nginx на `0.0.0.0:<порт>` перекрыл бы сервис
на `127.0.0.1:<порт>`.

---

> **Почему два файла, а не один:** гейт это отдельный `server{}` (в `conf.d/`), а
> редирект под путём это `location` **внутри** server-блока homeui (на 443, через
> `default.wb.d/`) — иначе ссылка не будет на origin homeui и login не сможет на
> неё вернуться. Это разные server-блоки, в один файл их не слить.

---

## Настройка

### Шаг 1. Поднять сервис на loopback

Сервис должен слушать **`127.0.0.1:<внутренний-порт>`** (например `127.0.0.1:9000`),
а не `0.0.0.0`. Если он торчит на `0.0.0.0` — порт обходит гейт, и защита
бессмысленна.

### Шаг 2. Проверить, что внешний порт свободен

```bash
ss -ltn | awk '{print $4}' | grep -E '[:.]29000$'   # пусто = порт 29000 свободен
```

### Шаг 3. Положить файл гейта

`/etc/nginx/conf.d/<сервис>.conf` — поменять 4 отмеченные строки:

```nginx
server {
    listen 29000 ssl;                              # (1) внешний порт (= 2+внутренний)
    server_name _;                                 #     на этом порту сервер один — матчит любой хост
    include /etc/nginx/snippets/wb-gate-tls.inc;

    location / {
        set $wb_role   admin;                      # (2) минимальная роль: admin | operator | user
        set $wb_return /open-my-service;           # (3) путь возврата (см. Шаг 4)
        auth_request /wb_auth_check;
        error_page 401 = @wb_unauth;
        proxy_pass http://127.0.0.1:9000;          # (4) внутренний адрес сервиса
        include /etc/nginx/snippets/wb-gate-proxy.inc;
    }
    include /etc/nginx/snippets/wb-gate-authcheck.inc;
    include /etc/nginx/snippets/wb-gate-unauth.inc;
}
```

Роль работает **«эта и выше»** (иерархия `user < operator < admin`): `user` —
любой залогиненный, `admin` — только администратор.

### Шаг 4. Положить редирект под путём (рекомендуется)

`/etc/nginx/includes/default.wb.d/<сервис>-bounce.conf` — одна строка, **порт в
ней должен совпасть** с `listen` из Шага 3:

```nginx
location = /open-my-service { return 302 https://$host:29000/; }
```

Зачем: после логина homeui разрешает возврат только на **свой origin** (другой
порт он отбрасывает как open-redirect), поэтому `$wb_return` указывает на этот
путь, а он уже 302-редиректит на порт. Плюс это готовая относительная ссылка без
хоста и порта.

Без этого файла гейт работает, но после логина пользователь окажется на дашборде
homeui, и порт придётся открыть руками.

### Шаг 5. Проверить и перезагрузить

```bash
nginx -t && systemctl reload nginx
```

**Если `nginx -t` ругается — исправить файл и не перезагружать.** Битый файл
уронит следующий reload (и веб-интерфейс заодно). Это на ответственности
владельца контроллера.

### Шаг 6 (опционально). Пункт в меню «Интеграции»

`/usr/share/wb-mqtt-homeui/custom-menu/<сервис>.json` — появляется без рестарта:

```json
{
  "id": "integrations",
  "children": [
    {
      "id": "my-service",
      "url": "/open-my-service",
      "isExternal": true,
      "openInNewTab": true,
      "title": { "ru": "Мой сервис", "en": "My Service" }
    }
  ]
}
```

`url` — тот же относительный путь из Шага 4 (не хардкодить хост/порт). Обновить
страницу homeui — пункт появится в «Интеграции».

## ⚠️ Обязательно проверить (иначе не заработает / небезопасно)

1. **Сервис слушает только `127.0.0.1`.** Иначе порт обходит гейт.
2. **Открывать сервис по ТОМУ ЖЕ hostname, что и homeui** (cloud-DNS
   `<ip>.<serial>.ip.wirenboard.com`). Сессионная кука host-only — на другом
   имени её не будет, и пойдёт бесконечный редирект.
3. **Прогнать `nginx -t`** перед reload (см. Шаг 5).
4. **Роль выбрать осознанно** — `user` пускает любого залогиненного.

## Как пользоваться

- Основной вход: открыть порт напрямую — `https://<хост>:29000/`.
- Удобный вход: ссылка `https://<хост>/open-my-service` (или пункт меню) →
  редиректит на порт.

## Ограничения (что НЕ работает)

- **Удалённый доступ через облачный тоннель Wiren Board.** Тоннель пробрасывает
  только основной веб-порт (443); доп. порт он не отдаёт, поэтому ни прямой вход,
  ни ссылка-редирект удалённо не сработают. Порт-гейт — для локальной сети /
  прямого доступа.
- **API-клиенты (скрипты).** Пока единственный способ — сессионная кука: клиент
  логинится в WB (`POST /auth/login` логином/паролем), получает куку `id` и шлёт
  её на запросы; гейт её проверит. Браузер получает редирект на форму, скрипт без
  куки — `401`. Отдельных API-токенов пока нет — следующая итерация.

## Что для этого несёт homeui (наша сторона, не пользователь)

1. **Лимит `/auth/check` поднят до 1000** (backend, `main.py`; `/auth/login`
   остаётся 30). Это убирает класс 429→500 при всплеске авторизованных
   подзапросов на открытии интерактивных сабаппов — без него на каждый сабапп
   пришлось бы вешать собственный кэш auth-check.

2. **Gate-сниппеты** в `/etc/nginx/snippets/`:

   `wb-gate-tls.inc`:
   ```nginx
   ssl_certificate     /etc/ssl/sslip.pem;
   ssl_certificate_key /etc/ssl/sslip.key;
   ssl_protocols TLSv1.2 TLSv1.3;
   ```

   `wb-gate-authcheck.inc` (требует, чтобы `location /` выставил `$wb_role`):
   ```nginx
   location = /wb_auth_check {
       internal;
       proxy_pass_request_body off;
       proxy_set_header Content-Length "";
       proxy_set_header Required-User-Type $wb_role;
       proxy_pass http://wb-homeui-back/auth/check;
   }
   ```

   `wb-gate-unauth.inc` (требует, чтобы `location /` выставил `$wb_return`). Разнесён
   с check, чтобы кэширующие гейты (напр. редактор Node-RED, чей всплеск бьёт даже
   лимит 1000) несли **свой** check, но переиспользовали этот фоллоу-ап как есть:
   ```nginx
   location @wb_unauth {
       # переход в браузере -> форма логина homeui; всё остальное -> 401
       if ($http_sec_fetch_mode = "navigate") {
           return 302 https://$host/login/?externalReturn=$wb_return;
       }
       return 401;
   }
   ```

   `wb-gate-proxy.inc` (внутри `location /`):
   ```nginx
   proxy_set_header X-WB-Role "";       # никогда не доверять клиентскому X-WB-*
   proxy_set_header X-WB-User "";
   proxy_set_header Host $host;
   proxy_set_header X-Real-IP $remote_addr;
   proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
   proxy_set_header X-Forwarded-Proto $scheme;
   proxy_http_version 1.1;
   proxy_set_header Upgrade $http_upgrade;
   proxy_set_header Connection "upgrade";
   ```

### Backlog (следующие итерации, не сейчас)

- **API-токены**: `/auth/check` учится принимать `Authorization: Bearer <token>`
  в дополнение к куке (хранилище токенов в БД homeui + выпуск/отзыв). Даёт
  машинным клиентам доступ без логин-флоу. Фича homeui, не твик.
- **Per-user identity** в сервисе: инъекция `X-WB-User`/`X-WB-Role` из вердикта
  `/auth/check` (homeui отдаёт роль заголовком `Wb-User-Type`). Это SSO-слой.
