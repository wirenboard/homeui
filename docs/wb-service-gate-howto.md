# Закрыть свой сервис авторизацией Wiren Board (порт-гейт)

Сервис живёт на отдельном порту; nginx пускает на него только залогиненных в homeui
с нужной ролью: браузер перекидывает на форму входа и возвращает обратно, API-клиент
получает `401`. Гейт **сам следует режиму HTTPS** homeui: переключили тумблер в
настройках — все гейты автоматически перенастроились на TLS или обычный HTTP.

## Настройка

1. Сервис должен слушать **только `127.0.0.1:<порт>`** — иначе внутренний порт
   обходит гейт.

2. Положить `/etc/wb-homeui/gates.d/<имя>.json` (имя — латиница/цифры/дефис):

   ```json
   {
     "internalPort": 9000,
     "externalPort": 29000,
     "role": "admin",
     "menu": { "title": { "ru": "Мой сервис", "en": "My Service" } }
   }
   ```

   | Поле | Обязательное | Что делает |
   |---|---|---|
   | `externalPort` | да | внешний порт гейта — по нему сервис доступен снаружи. Удобно брать внутренний порт с «2» спереди (`9000 → 29000`). Не должен совпадать с портами других гейтов и их сервисов |
   | `internalPort` | да | порт сервиса на `127.0.0.1`; не должен совпадать с внешним — иначе гейт перекрыл бы сам сервис |
   | `role` | нет (`admin`) | минимальная роль, «эта и выше»: `user < operator < admin`; `user` пускает любого залогиненного |
   | `auth` | нет (`true`) | `false` — пустить **без авторизации**: тот же единый порт-механизм, но без формы входа (доверенные `X-WB-*` всё равно зачищаются) |
   | `menu` | нет | пункт в меню «Интеграции»; без `menu` пункт не создаётся. `menu.title` — подписи `ru`/`en`; **достаточно одного языка — для второго покажется он же**. У ролей ниже `role` пункт скрыт, но это только видимость — доступ контролирует сам гейт |

   JSON Schema формата — `/usr/share/wb-mqtt-homeui/wb-gate.schema.json`
   (подключите в редакторе для подсказок и валидации).

3. Проверить и применить:

   ```bash
   wb-homeui-gates check   # что отрендерится; битые файлы с причинами
   wb-homeui-gates apply   # рендер + nginx -t + reload nginx
   ```

   Перезапускать ничего не нужно. `apply` при ошибке nginx откатывает к прежнему
   рабочему состоянию и печатает причину — сломать nginx или соседние гейты нельзя.
   Backend сам переприменяет гейты при старте и при переключении тумблера HTTPS.

## Как пользоваться

- Через меню или `<хост>/open-<имя>` — редирект на порт гейта.
- Открывать по **тому же hostname, что и homeui** — сессионная кука host-only,
  на другом имени будет бесконечный редирект на логин.

## Как это устроено

- Конфиги рендерятся в `/var/lib/wb-homeui/nginx-gates/` (не редактировать руками);
  режим TLS/HTTP определяется тумблером HTTPS и наличием пригодного сертификата,
  меняется вместе с ними.
- `/etc/wb-homeui` (`gates.d` и `custom-menu`) сохраняется при обновлении прошивки
  через wb-configs — симлинк в `/mnt/data` появляется после перезагрузки.
- Ответы `/auth/check` кэшируются на 5 секунд — шторм запросов от активного SPA
  за гейтом не упирается в лимит backend'а.
- Нужны нестандартные nginx-директивы (таймауты, свои location)? Положите рядом
  `<имя>.nginx.inc` — он включается в server-блок гейта. Ошибку в нём поймает
  `nginx -t` при `apply`, применение откатится.
- TLS включается только когда сертификат реально получен и пригоден; до этого
  (и если сертификат пропал/побился) гейты и основной UI работают по HTTP —
  backend вернёт HTTPS сам, как только сертификат появится.

## Ограничения

- **Облачный тоннель Wiren Board** пробрасывает только основной веб-порт —
  порт-гейт работает в локальной сети / при прямом доступе.
- **API-клиенты (скрипты)**: логин `POST /auth/login`, дальше кука `id` с каждым
  запросом; без куки — `401`.
- Занятость внешнего порта другим приложением `nginx -t` не ловит — если гейт не
  отвечает после `apply`, проверьте `ss -ltn` и журнал nginx.
- Запросы к гейту лимитируются по IP клиента (15 r/s, всплеск до 200) — устойчивое
  превышение даёт `429`. Обычному UI за гейтом этого хватает с запасом.
